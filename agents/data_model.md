# Data Model

## Purpose

Define canonical models for the Flutter rewrite.

`agents/rules.md` describes the current Python data shape.
`agents/rules_fixes.md` defines target corrections.

The rewrite must keep behavior precise while enforcing safer model boundaries.

## Day System

The system is a five-day weekday roster.

Canonical day order:

```text
0: PAZARTESİ
1: SALI
2: ÇARŞAMBA
3: PERŞEMBE
4: CUMA
```

Display labels may use Turkish characters:

```text
Pazartesi
Salı
Çarşamba
Perşembe
Cuma
```

Implementation must centralize the exact display strings and canonical uppercase strings in one day model.

No code outside the day model may hard-code:

- `5`
- `6`
- weekday column indexes
- weekday names
- Excel B-F assumptions
- PDF six-column assumptions

## Week Model

Canonical Week fields:

```text
Week
- title: String
- startDate: Date
- endDate: Date
- schoolName: String
- principalName: String
- locations: List<String>
- roster: Roster
```

Rules:

- `title` is generated by `WeekService.buildTitle` unless importing an existing source title.
- `startDate` and `endDate` must be real dates.
- `schoolName` and `principalName` preserve display spelling after whitespace cleanup.
- `locations.length` must equal `roster.rows.length`.
- A Week may be an exportable week only when `title.trim()` is non-empty.
- Blank placeholder weeks are UI-only unless a later explicit export rule allows them.

## Roster Model

Canonical Roster:

```text
Roster
- rows: List<RosterRow>
```

Canonical RosterRow:

```text
RosterRow
- location: String
- teachersByDay: List<String> length 5
```

The legacy Python shape separates:

```text
locations: List<String>
roster: List<List<String>>
```

Flutter may expose that shape only at import/export mapping boundaries.
Inside the app, prefer `RosterRow` so location and teacher cells cannot drift apart.

## Normalized Roster Rows

Every roster row must be normalized to exactly five teacher cells.

Normalization:

- Convert `null` row to five empty strings.
- Pad rows shorter than five with empty strings.
- Trim rows longer than five to Friday.
- Convert `null` teacher cell to empty string.
- Collapse repeated whitespace inside teacher names.
- Trim leading/trailing whitespace.
- Preserve user-entered display case.

This implements `rules_fixes.md` FIX 8 and FIX 9.

## Location Rules

Location text:

- must be cleaned with display normalization
- must not be empty when any teacher cell in the same row is non-empty
- may be non-empty when all teacher cells are empty

Duplicate-location comparison:

- must use the Turkish-aware canonical text function
- must remove Unicode whitespace and supported dash characters
- must not remove all punctuation by default

Duplicate-location runs:

- must follow the explicit duplicate-run policy from `rules_fixes.md` FIX 7
- must not silently merge only the first pair of a longer duplicate run

## DateRange Model

Canonical DateRange:

```text
DateRange
- start: Date
- end: Date
```

Validation:

- start is required
- end is required
- start must be <= end
- standard roster weeks are five calendar days: start through start + 4 days
- previous/next week generation moves both dates by 7 days

Cross-year parsing:

- If a parsed title has `endMonth < startMonth` and no explicit end year, end year must be start year + 1.

## ExportSnapshot Model

ExportSnapshot:

```text
ExportSnapshot
- weeks: List<Week>
- source: currentWeek | generatedPreview | explicitPreview
- metadataScope: file | pageGroup | week
```

Rules:

- The preview must render exactly the ExportSnapshot weeks.
- PDF export must use exactly the ExportSnapshot weeks.
- Excel export must use exactly the ExportSnapshot weeks.
- Editable preview changes must update the ExportSnapshot or preview editing must be disabled.
- Valid-week filtering must be shared by PDF and Excel.

## MergePlan Model

MergePlan describes visual row/column merges without mutating the logical roster.

Fields:

```text
MergePlan
- rowSpans: List<RowSpan>
- cellSpans: List<CellSpan>
- conflicts: List<MergeConflict>
```

Rules:

- MergePlan is generated from normalized RosterRows.
- Logical data remains intact.
- Excel and PDF renderers consume the same MergePlan.
- Excel import must be merge-aware if reading visually merged files.

## Validation Rules

Week validation:

- title is required for export
- startDate is required
- endDate is required
- startDate <= endDate
- locations and roster rows must have equal lengths
- each roster row has exactly five teacher cells
- no teacher-filled row may have an empty location

Roster validation:

- row count may be zero for draft state
- row count must be non-zero before generation/export
- each row location is display-normalized
- each teacher cell is display-normalized
- no row may contain more than five logical teacher cells after normalization

Export validation:

- export snapshot must contain at least one valid week
- all weeks must use normalized roster rows
- PDF and Excel valid-week filtering must produce the same count
- blank placeholders must not export unless explicitly requested by a future rule

Import validation:

- title row must be detected using Turkish-aware canonical text
- weekday headers must be detected using the centralized day model
- imported roster rows must be normalized
- merged Excel cells must be resolved before empty-row detection

## Model Immutability

Domain models should be immutable.

State changes should create new model instances.

Allowed mutation:

- temporary edit drafts inside a ViewModel
- low-level parser buffers inside services

Immutable models reduce drift between preview, current week, and export snapshots.
